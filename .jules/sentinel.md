## 2025-02-17 - [CTE Shadowing Vulnerability in Table Extraction]
**Vulnerability:** A critical security vulnerability was found where `extractAllTables` improperly converted all table names to lowercase using `.toLowerCase()` when building the `cteScope` set. In PostgreSQL, quoted identifiers (e.g., `"SECRET_TABLE"`) preserve their case and are treated differently from unquoted identifiers (which implicitly map to lowercase). An attacker could create a CTE using a quoted uppercase name (e.g. `"SECRET_TABLE"`) that shadowed a real table with the corresponding lowercase name (e.g. `secret_table`), effectively bypassing the policy validation logic for the real table.
**Learning:** SQL identifier casing is tricky and SQL-dialect dependent. When extracting relations and managing scope (like CTEs), normalisation must preserve casing as defined by the dialect's rules (PostgreSQL preserves case in double quotes).
**Prevention:** Ensure any normalisation of SQL identifiers explicitly respects quoted semantics. In PostgreSQL, double quotes define case-sensitive matching boundaries and these must not be blindly lowercased.

## 2025-03-18 - [Runtime Type Confusion in Public API]
**Vulnerability:** The public `validate` API accepted the `sql` parameter typed as `string`, but did not perform runtime type validation. When consumed from pure JavaScript environments without TypeScript compilation enforcement, an attacker or flawed application logic could pass non-string values (like `null`, `undefined`, or complex objects). This resulted in unhandled runtime exceptions (`TypeError: null is not an object (evaluating 'sql.length')`), which could lead to Denial of Service (DoS) or application crashes.
**Learning:** Type confusion vulnerabilities are a major risk for libraries operating at security boundaries. TypeScript types are erased at runtime; an explicitly typed parameter (e.g. `sql: string`) does not guarantee a string at execution time.
**Prevention:** Always implement explicit runtime type checks (`typeof input === 'string'`) at public API boundaries, especially for security-critical validation libraries, rather than trusting TypeScript typings alone.

## 2025-03-20 - [Runtime Type Confusion in allowMultiStatement and resolver]
**Vulnerability:** The public API `validate` function in `sql-guard` lacked explicit runtime type validation for boolean properties (`allowMultiStatement`) and function properties (`resolver`) in the `policy` object. An attacker or flawed application logic could bypass the multi-statement protection by passing a truthy string (e.g., `"false"`) for `allowMultiStatement`, or cause unhandled exceptions leading to denial-of-service by passing a string to `resolver`.
**Learning:** Boolean checks like `!allowMulti` will fail open if `allowMulti` is a truthy string like `"false"`. In JavaScript, TypeScript types are erased, so library inputs must be rigidly typed at runtime.
**Prevention:** Explicitly validate types of configuration options (e.g., `typeof policy.allowMultiStatement !== 'boolean'`, `typeof policy.resolver !== 'function'`) at the entry boundary before using them in validation logic.
## 2025-05-18 - [Shadowing Vulnerability in Table AST Deduplication]
**Vulnerability:** The AST table deduplication logic used `.toLowerCase()` on both `schema` and `name` to generate keys for uniqueness, which stripped away case variations that the node-sql-parser previously preserved (such as unquoted identifiers versus double-quoted identifiers). A malicious user could intentionally inject `SELECT * FROM "SECRET_TABLE", secret_table` causing the uppercase and lowercase variant to look identical and the second identifier dropping from the AST list before policy validation check.
**Learning:** Node-sql-parser removes double quotes from identifiers but retains case. PostgreSQL matches unquoted inputs case-insensitively but quoted inputs case-sensitively. Lowercasing AST entities before verifying bypasses this dialect distinction.
**Prevention:** Always deduplicate string identifiers using exact matches on raw AST data to prevent accidental identifier folding that enables shadowing bypass.
## 2025-03-21 - [Deduplication Shadowing Vulnerability in Table Extraction]
**Vulnerability:** A vulnerability was discovered in `extractAllTables` where the AST table deduplication process incorrectly folded table identifiers (schema and name) to lowercase using `.toLowerCase()`. Since `node-sql-parser` preserves casing for double-quoted identifiers in its AST (e.g. `"SECRET_TABLE"` stays `SECRET_TABLE`) but unquoted ones are case-insensitive in PostgreSQL logic, this caused a quoted identifier `"SECRET_TABLE"` to shadow an unquoted identifier `secret_table`. An attacker could exploit this by crafting a query mixing case-sensitive and case-insensitive table references, allowing unauthorized access to tables to bypass policy restrictions.
**Learning:** In SQL-based policy validation using AST parsing, identifier normalization for structural tasks (like deduplication) must exactly match the semantics of the AST output before policy normalisation happens. `node-sql-parser` keeps case for quoted and strips quotes.
**Prevention:** Never arbitrarily alter case (`.toLowerCase()`) when performing deduplication on AST properties that rely on specific casing rules for correctness.
## 2025-05-18 - [Type Confusion DOS via Unvalidated Array Elements in Public API]
**Vulnerability:** While `compilePolicy` checked if `policy.allowedTables`, `policy.allowedFunctions`, and `policy.allowedStatements` were arrays, it failed to validate the runtime type of the individual elements within those arrays. If a pure-JS consumer provided an object without a valid `toString` method (like `Object.create(null)`) or a non-string type, the policy compiler's use of template literals (`` `${String(table)}` ``) or `.trim()` would throw an unhandled `TypeError` (e.g. "Cannot convert object to primitive value"), bypassing the validation catch blocks and resulting in an application crash / DoS.
**Learning:** Runtime type verification must be deep. Checking `Array.isArray()` is insufficient if the code subsequently assumes every element of the array conforms to the TypeScript type `string[]`.
**Prevention:** Always iterate and assert the expected type of every element in an array parameter at the public API boundary before using it in string formatting or logic.
## 2026-04-17 - [Safe String Utility for DoS Prevention]
**Vulnerability:** Unhandled exceptions causing DoS vulnerabilities via objects with malicious `toString` methods or throwing getters during string conversions.
**Learning:** Even internal mechanisms like type extraction or error message parsing can act as an attack vector if untrusted user inputs (like maliciously formed objects passed in policies or mocked ASTs) define a malicious `toString` method, leading to unhandled exceptions and breaking validation logic.
